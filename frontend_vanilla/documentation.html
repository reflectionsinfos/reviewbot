How ReviewBot Works

Understanding the autonomous review architecture, strategies, and decision-making process.

Overview

ReviewBot uses a multi-strategy analysis system to review your codebase against structured checklists. Instead of using a single approach for all questions, ReviewBot intelligently selects the best analysis method for each checklist item based on what the question is asking.

💡 Why Multiple Strategies?

Using the right tool for each question improves accuracy, reduces cost, and speeds up review time. Simple file checks cost $0 and take milliseconds, while complex architectural questions benefit from LLM analysis.

Architecture

1. User Initiates Review
Via VS Code Plugin or CLI • Selects project path and checklist
↓
2. Strategy Decision (LLM)
Classifies each item: file_presence, pattern_scan, llm_analysis, human_required, etc.
↓
3. Analyzer Execution
Runs the appropriate analyzer based on strategy
↓
4. Results Stored
RAG status, evidence, confidence, files checked
↓
5. Report Generated
Compliance score, gaps, recommendations, action items
↓
6. Action Plan Generated
Prioritised action cards (Critical / Advisory / Sign-off), AI prompts per finding (Generic / Cursor / Claude Code)

Review Strategies

ReviewBot supports 6 different analysis strategies. The strategy router automatically selects the most appropriate one for each checklist item.

Strategy What It Does When Used Cost Time
file_presence Checks if specific files or directories exist "Is there an ARCHITECTURE.md?" Free ~10ms
pattern_scan Regex scan of source files for patterns "Are there hardcoded secrets?" Free ~50ms
metadata_check Checks package.json, requirements.txt, etc. "Are dependencies scanned for vulnerabilities?" Free ~20ms
llm_analysis LLM analyzes code structure and quality "Is the code well-architected?" ~$0.002-0.01 ~3-15s
human_required Skipped for AI — requires human interview "What's team morale?" Human time —
ai_and_human AI analyzes first, human validates "Is the architecture sound?" ~$0.002 + human ~3-15s + review

How Strategy is Decided

Current Implementation (LLM-Based Classification)

ReviewBot uses an LLM to semantically classify each checklist item into the appropriate strategy. This is more accurate than keyword matching and handles variations in question phrasing.

🧠 Example Classification
Input: "Do you have design documentation?"
Output: strategy = "file_presence"
        file_patterns = ["DESIGN.md", "design.md", "docs/design/"]

Input: "Are credentials stored securely?"
Output: strategy = "pattern_scan"
        scan_patterns = [r"password\s*=\s*['\"].{6,}['\""]

Input: "Is the codebase well-structured?"
Output: strategy = "llm_analysis"
        context_keywords = ["architecture", "structure", "modularity"]

Decision Priority

  1. Database Rules — Admin-defined overrides (ChecklistRoutingRule table)
  2. Human Required Detection — Organizational/process questions (fast path)
  3. LLM Classification — Semantic analysis of area + question
  4. Default — llm_analysis (if uncertain)

Analyzers

Each strategy has a dedicated analyzer module that executes the actual review logic.

FilePresenceAnalyzer

# Checks for files/directories matching patterns
found = file_index.find_files(["ARCHITECTURE.md", "*.drawio"])
if found:
    return AnalysisResult(rag_status="green", evidence=f"Found {len(found)} files")
else:
    return AnalysisResult(rag_status="red", evidence="No matching files found")

PatternScanAnalyzer

# Regex scan of source files
matches = []
for file in source_files:
    if re.search(r"password\s*=\s*['\"]", file.content):
        matches.append(file.path)

if matches:
    return AnalysisResult(rag_status="red", evidence=f"Found secrets in {len(matches)} files")

LLMAnalyzer

# Sends code context to LLM for analysis
prompt = build_prompt(checklist_item, code_context)
response = llm_client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": prompt}]
)
return parse_llm_response(response)

MetadataCheckAnalyzer

# Checks package.json, requirements.txt, etc.
deps = parse_package_json("package.json")
vulnerabilities = scan_dependencies(deps)
if vulnerabilities:
    return AnalysisResult(rag_status="amber", evidence=f"{len(vulnerabilities)} vulnerable deps")

Results Storage

Each review result is stored in the autonomous_review_results table:

Column Type Description
strategy String Which strategy was used (file_presence, llm_analysis, etc.)
rag_status String green | amber | red | na | skipped
evidence Text What was found (file list, LLM reasoning, etc.)
confidence Float 0.0-1.0 (1.0 for deterministic, 0.7-0.9 for LLM)
files_checked JSON List of file paths examined
skip_reason Text Why item was skipped (human_required)
needs_human_sign_off Boolean AI ran but human must confirm

Manual Strategy Override

Admin users can override the automatic strategy decision by creating a routing rule:

-- Via API
POST /api/routing-rules
{
  "checklist_item_id": 42,
  "strategy": "human_required",
  "skip_reason": "This requires organizational data — interview the PM",
  "evidence_hint": "Provide roadmap document or stakeholder interview notes"
}

Database rules take priority over automatic classification, allowing fine-tuned control for edge cases.

Checklist Item Metadata

Each checklist item includes workflow metadata that affects scoring and review completion.

Field Meaning Current Intent
weight Relative importance of the checklist item in weighted scoring. Higher weight means the item contributes more to the compliance score.
is_review_mandatory Whether the item is mandatory for review completion. If true, the review should not be considered complete until the item has been reviewed and any required human sign-off is done.
Review Completion vs Compliance

A mandatory item must be reviewed before the review can be completed. It does not have to be green. Compliance still depends on the final outcome of reviewed items.

Action Plan — From Findings to Fixes

Once an autonomous review completes, ReviewBot can generate a structured Action Plan — a prioritised list of remediation tasks derived directly from the review findings. Each task includes not just what failed, but what to do about it, and an AI-ready prompt you can paste straight into your IDE assistant.

💡 Why an Action Plan?

A compliance report tells you what is wrong. An action plan tells you exactly what to do next. Without it, developers still have to interpret raw findings, decide priority, and manually write prompts for their AI tools. ReviewBot eliminates that translation work — the gap between "found" and "fixed" is reduced to a single copy-paste.

The Problem It Solves

After a review, a team typically has three friction points before remediation can begin:

  1. Prioritisation — Which findings are blockers vs. nice-to-haves?
  2. Interpretation — The AI evidence is verbose; what is the concrete task?
  3. AI IDE setup — Developers need to manually craft prompts for Cursor, GitHub Copilot Chat, or Claude Code to fix each issue.

The Action Plan addresses all three in a single step, automatically, as soon as the review job completes.

Core Logic

The ActionPlanGenerator service runs entirely server-side and requires no additional LLM calls for the base plan. It processes the stored review results using deterministic rules:

1. Load completed review results
AutonomousReviewResult rows for the job — rag_status, evidence, files_checked, needs_human_sign_off
↓
2. Categorise by RAG status
red → Critical Blocker  |  amber → Advisory  |  needs_human_sign_off=true → Sign-off Required  |  green → Compliant
↓
3. Derive remediation instruction
If expected_evidence set → "Implement: {expected_evidence}"  |  else → "Address: {question}"
↓
4. Build AI prompts (3 flavours)
Generic  |  Cursor / Copilot Chat (@workspace prefix)  |  Claude Code (Task: prefix)
↓
5. Return structured ActionPlanResponse
JSON with critical_blockers, advisories, sign_off_required, compliant_summary, summary stats

Action Card Structure

Each finding becomes an Action Card with the following fields:

Field Source Purpose
priority Derived from rag_status High (red) or Medium (amber) — drives developer triage
what_was_found AutonomousReviewResult.evidence Verbatim AI finding — what the analyzer observed in the codebase
what_to_fix Derived from ChecklistItem.expected_evidence Single imperative sentence — exactly what the developer must implement
expected_outcome ChecklistItem.expected_evidence The acceptance criterion — what "done" looks like for this item
ai_prompt Generated by ActionPlanGenerator Ready-to-use prompt in 3 IDE flavours (see below)

AI Prompt Flavours

Each Action Card contains three variants of the same prompt, tuned for different AI developer tools:

Flavour Prefix Target Tool Extra Instruction
Generic None Any AI assistant (ChatGPT, Gemini, etc.) Plain context + task + acceptance criteria
Cursor / Copilot @workspace Cursor IDE, GitHub Copilot Chat "Search the workspace for related files before making changes"
Claude Code Task: Claude Code CLI "Run tests after making changes"

Every prompt contains four structured blocks — CONTEXT (project name, tech stack, files examined), FINDING (what the AI found), STANDARD EXPECTED (the checklist requirement), and TASK + ACCEPTANCE CRITERIA — ensuring the developer assistant has complete context without the developer writing a single word.

📋 Example Prompt (Generic flavour)
CONTEXT:
Project: Hatch Pay
Tech Stack: Python, FastAPI, PostgreSQL, Docker
Files Checked:
  - src/auth/middleware.py
  - src/api/routes/auth.py

FINDING:
No rate limiting found on authentication endpoints.
Login and token refresh routes accept unlimited requests.

STANDARD EXPECTED:
Rate limiting must be applied to all authentication endpoints
(max 5 requests/min per IP for login).

TASK:
Fix the above gap. Ensure the solution satisfies:
'Are authentication endpoints protected by rate limiting?'

ACCEPTANCE CRITERIA:
Rate limiting must be applied to all authentication endpoints
(max 5 requests/min per IP for login).

Sign-off Required Category

Items where needs_human_sign_off = true are collected into a separate Sign-off Required bucket, regardless of their RAG status. This handles findings where the AI ran successfully but the business stakes are high enough that a human must confirm the result before marking it resolved — for example, data retention policies, compliance attestations, or architectural sign-off.

LLM Enhancement (Optional)

The base Action Plan requires no LLM calls and is instant. For deeper, codebase-specific prompts, users can trigger the Enhance with AI button in the History view:

POST /api/autonomous-reviews/{job_id}/action-plan/enhance
Triggers LLM enrichment for all red and amber results
↓
LLM receives: question + evidence + expected_evidence + files_checked + project tech stack
System prompt: "You are a senior engineer writing actionable remediation prompts for AI IDEs..."
↓
Returns enhanced JSON
specific_instruction, files_to_modify, acceptance_criteria — stored in job.agent_metadata["action_plan_prompts"]

Enhancement is idempotent — calling enhance twice returns the cached result without a second LLM call. Enhanced prompts are stored on the job record so they persist across sessions.

Advantages Over Manual Remediation Planning

Without Action Plan With Action Plan
Developer reads verbose AI evidence and interprets what needs to change Single imperative sentence: "Implement: {expected_evidence}"
Team manually sorts findings by severity in a spreadsheet Pre-sorted into Critical Blockers / Advisories / Sign-off Required
Developer writes a custom prompt for their AI IDE from scratch Three flavoured prompts ready to copy-paste (Generic / Cursor / Claude Code)
Context (tech stack, files examined) must be re-stated in every prompt Context auto-injected from review metadata — zero manual effort
Human sign-off items mixed in with automated findings Clearly separated bucket with explicit human action required
No exportable artefact for project management One-click Markdown export for tickets, Confluence, GitHub issues

API Endpoints

GET  /api/autonomous-reviews/{job_id}/action-plan
     → Returns full ActionPlanResponse (instant, no LLM)
     → 400 if job not in "completed" status
     → 404 if job not found

POST /api/autonomous-reviews/{job_id}/action-plan/enhance
     → Triggers LLM enrichment for red/amber items
     → 200 {"status":"already_enhanced"} if run before (idempotent)
     → 200 {"status":"enhanced","prompts_generated":N} on success

How to Use It

  1. Complete an autonomous review (status must be completed).
  2. Open History and click the job row.
  3. Switch to the Action Plan tab.
  4. Choose your IDE flavour from the dropdown (Generic, Cursor/Copilot, or Claude Code) — persisted in localStorage.
  5. Expand any card, click Show AI Prompt, then Copy.
  6. Paste directly into your AI IDE assistant.
  7. Optionally click Enhance with AI for deeper, file-aware prompts.
  8. Click Export MD to download the full plan as a Markdown file for your team.

Local Review Tools

For a seamless developer experience, ReviewBot provides two local tools that allow you to run reviews directly from your machine or CI/CD pipelines without manually entering repository URLs.

🛡️ VS Code Extension

The ReviewBot VS Code extension integrates autonomous reviews directly into your IDE. It provides real-time feedback and visual decorations on your source code.

Key Features

  • Gutter Decorations: Instant R/A/G icons next to your code lines indicating compliance status.
  • Sidebar Results: A dedicated view for tracking review progress and viewing detailed evidence.
  • Direct Login: Uses standard JWT authentication to sync with your ReviewBot account.
  • Context Injection: Automatically detects your open workspace and tech stack.

Installation

  1. Open the AI Review page to download the latest .vsix package.
  2. In VS Code, go to the Extensions view (Ctrl+Shift+X).
  3. Click the ... menu and select Install from VSIX....
  4. Select the downloaded file and restart VS Code.

AI streaming results directly to the VS Code sidebar.

💻 CLI Agent

The ReviewBot CLI is a lightweight Python-based scanner designed for headless environments, CI/CD, and power users who prefer the terminal.

Installation

# Extract the reviewbot-agent.zip then:
pip install .

Common Commands

Command Description
reviewbot login Authenticate with the server
reviewbot status Show current login and server info
reviewbot review Scan directory and run a review
reviewbot logout Clear local credentials

Example Usage

# Basic review with output file
reviewbot review \
  --project-id 5 \
  --checklist-id 11 \
  --source-path ./my-service \
  --output ./report.md
💡 CI/CD Integration

You can automate reviews in your pipeline by setting the REVIEWBOT_SERVER and REVIEWBOT_API_KEY environment variables, allowing the agent to run without an interactive login.
